# 工作流程规则 - 超级个体开发团队

## 🎯 严格Review流程

### 每个阶段强制Review节点

#### 1. 需求分析阶段
- **负责人**: 产品经理
- **Review检查点**:
  - ✅ 市场分析完整性
  - ✅ 竞品分析深度
  - ✅ 经济可行性评估
  - ✅ 用户故事清晰度
- **输出文档**:
  - `docs/01-市场分析报告.md`
  - `docs/02-用户故事.md`
  - `docs/03-MVP计划.md`

#### 2. 界面设计阶段
- **负责人**: UI-X设计师
- **Review检查点**:
  - ✅ HTML原型完整性
  - ✅ 响应式设计效果
  - ✅ 交互逻辑合理性
  - ✅ 跨平台兼容性
- **输出文档**:
  - `prototype.html`
  - `docs/04-设计系统.md`
  - `assets/css/style-guide.css`

#### 3. 架构设计阶段
- **负责人**: 系统架构师
- **Review检查点**:
  - ✅ 技术选型合理性（开源社区前3）
  - ✅ 架构可扩展性
  - ✅ 性能优化方案
  - ✅ API规范完整性
- **输出文档**:
  - `docs/05-技术选型报告.md`
  - `docs/06-系统架构设计.md`
  - `docs/07-API规范.md`

#### 4. 前端开发阶段
- **技术栈优先级**: Vue > React > Angular
- **Review检查点**:
  - ✅ 代码规范符合标准
  - ✅ 组件化程度
  - ✅ 性能优化实现
  - ✅ 测试覆盖率
- **输出代码**:
  - 按技术栈: `src/` 目录
  - 构建配置: `vite.config.js` / `webpack.config.js`

#### 5. 后端开发阶段
- **技术栈**: FastAPI > Node.js
- **Review检查点**:
  - ✅ API实现完整性
  - ✅ 数据库设计合理性
  - ✅ 安全措施到位
  - ✅ 性能测试结果
- **输出代码**:
  - FastAPI: `app/` 目录
  - Node.js: `server/` 目录

#### 6. 测试阶段
- **负责人**: 测试工程师
- **Review检查点**:
  - ✅ 测试用例覆盖率
  - ✅ 边界条件测试
  - ✅ 性能测试达标
  - ✅ 安全测试通过
- **输出文档**:
  - `docs/08-测试计划.md`
  - `tests/test-cases.md`
  - `docs/09-测试报告.md`

#### 7. 部署阶段
- **负责人**: 运维工程师
- **Review检查点**:
  - ✅ Docker配置正确性
  - ✅ CI/CD流程完整性
  - ✅ 监控告警配置
  - ✅ 回滚机制验证
- **输出配置**:
  - `Dockerfile`
  - `docker-compose.yml`
  - `.github/workflows/deploy.yml`
  - `docs/10-部署指南.md`

#### 8. 文档整理阶段
- **负责人**: 技术文档工程师
- **Review检查点**:
  - ✅ 文档完整性
  - ✅ 格式规范性
  - ✅ 准确性验证
  - ✅ 用户友好性
- **输出文档**:
  - `docs/11-API文档.md`
  - `docs/12-用户手册.md`
  - `README.md`
  - `CHANGELOG.md`

## 📋 Review执行标准

### Review流程
1. **阶段完成** → 2. **自检清单** → 3. **主人Review** → 4. **问题修复** → 5. **确认通过**

### 文档命名规范
```
{阶段序号}-{内容描述}.md
例如：
01-市场分析报告.md
02-用户故事.md
03-MVP计划.md
```

### 文档标准格式
```markdown
# 文档标题

## 概述
简要说明文档目的

## 实现细节
详细技术实现

## 测试要点
需要重点测试的内容

## Review记录
- Review时间: YYYY-MM-DD
- Review人: 主人
- 问题清单: 
  - [ ] 问题1
  - [ ] 问题2
- 修改建议:
  - 建议1
  - 建议2
```

## 🔄 阶段间输入输出

每个阶段的输出作为下一阶段的输入：
- 需求文档 → 设计依据
- 设计原型 → 开发参考
- 架构设计 → 技术规范
- 测试报告 → 上线依据
- 部署配置 → 运维手册

## ⚡ 技术选型原则

### 开源社区前3标准
1. **GitHub Stars**: > 10k
2. **社区活跃度**: 最近更新 < 1个月
3. **使用案例**: 知名企业采用

### 技术栈优先级
- **前端**: Vue.js > React > Angular
- **后端**: FastAPI > Node.js
- **数据库**: PostgreSQL > MySQL > SQLite
- **部署**: Docker > Kubernetes > PM2
