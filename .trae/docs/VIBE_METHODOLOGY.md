# 🚀 VIBE 开发方法论

> **Documentation → Interface → Backend → Everything**

---

## 🎯 什么是 VIBE？

VIBE = **D**ocumentation + **I**nterface + **B**ackend + **E**verything

是一种**文档驱动**的迭代开发方法，强调：
- 📝 **文档先行** - 先写清楚要做什么
- 🎨 **前后端对齐** - 接口定义优先
- 🧪 **测试保障** - 自动化测试覆盖
- 🔄 **持续迭代** - 小步快跑，快速反馈

---

## 📋 VIBE 开发流程

### Phase 1: 📝 Documentation (文档驱动)

**目标**: 明确需求，建立共识

**任务**:
1. ✅ 讨论并确认项目需求
2. ✅ 创建产品需求文档 (PRD)
3. ✅ 定义功能清单和优先级
4. ✅ 识别技术风险和依赖

**产出**:
- `docs/PRD.md` - 产品需求文档
- `docs/FEATURES.md` - 功能清单
- `docs/TECH_STACK.md` - 技术选型

**检查点**:
- [ ] 需求是否清晰无歧义？
- [ ] 所有干系人是否达成共识？
- [ ] 技术风险是否识别？

---

### Phase 2: 🎨 Interface (前后端对齐)

**目标**: 统一接口，减少返工

**任务**:
1. ✅ 设计 API 接口规范
2. ✅ 定义数据模型
3. ✅ 制定错误码规范
4. ✅ 前后端接口评审

**产出**:
- `docs/API.md` - API 文档
- `docs/DATA_MODEL.md` - 数据模型
- `docs/ERROR_CODES.md` - 错误码规范

**检查点**:
- [ ] API 是否符合 RESTful 规范？
- [ ] 数据模型是否完整？
- [ ] 前后端是否评审通过？

---

### Phase 3: 🧪 Backend (分层开发)

**目标**: 测试先行，保证质量

**任务**:
1. ✅ 编写单元测试
2. ✅ 实现核心业务逻辑
3. ✅ 数据库设计和实现
4. ✅ API 接口实现

**产出**:
- `tests/` - 测试文件
- `app/` 或 `src/` - 业务代码
- `docs/API_IMPL.md` - 接口实现文档

**检查点**:
- [ ] 测试覆盖率 > 80%？
- [ ] 核心逻辑是否测试通过？
- [ ] API 是否符合接口规范？

---

### Phase 4: 🎨 Frontend (前端实现)

**目标**: 快速迭代，及时反馈

**任务**:
1. ✅ 实现 UI 组件
2. ✅ 对接 API
3. ✅ 集成测试
4. ✅ 用户体验优化

**产出**:
- `src/` 或 `app/` - 前端代码
- `tests/` - 前端测试

**检查点**:
- [ ] UI 是否符合设计稿？
- [ ] API 对接是否正常？
- [ ] 用户体验是否流畅？

---

### Phase 5: 🔄 Iteration (持续迭代)

**目标**: 快速修复，持续优化

**任务**:
1. ✅ 收集用户反馈
2. ✅ 修复 Bug
3. ✅ 性能优化
4. ✅ 功能迭代

**循环**:
```
需求变更 → Documentation → Interface → Backend → Frontend → 测试 → 上线
```

## 🎯 VIBE 核心原则

### 1. 文档驱动
> "写清楚再动手，避免返工"

### 2. 接口优先
> "前后端对齐，减少等待"

### 3. 测试保障
> "没有测试的代码是债务"

### 4. 小步快跑
> "快速迭代，持续反馈"

### 5. 质量内建
> "质量是过程，不是结果"

## 📝 VIBE 命令

在控制台中使用：

```
# 启动 VIBE 开发流程
python .trae/workflows/trae-console.py
→ 选择: VIBE 开发

# VIBE 快速命令
vibe init      # 初始化 VIBE 项目
vibe doc       # 生成文档模板
vibe api       # 生成 API 模板
vibe test      # 生成测试模板
```
